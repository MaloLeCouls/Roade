# 03 — Matrice de profondeur des blocs (P10)

> **Cœur de P10.** Un seul grand tableau : chaque ligne est un bloc, chaque colonne une capacité du « socle commun ». L'objectif est de matérialiser les dissymétries de profondeur — l'écart entre ce qu'un bloc *aurait* à offrir au titre du contrat implicite, et ce qu'il *offre* réellement. Toutes les preuves renvoient à `fichier:ligne` ; les définitions de chaque bloc sont dans `02_BLOCK_CATALOG.md`.

## Définition des capacités auditées

1. **UI de config dédiée** — l'Inspector a un composant `*Config` propre au bloc (au-delà de `label` / `description` / `locked`).
2. **Sélecteur de colonnes typé** — quand l'Inspector propose de choisir une colonne, le **type** est affiché à côté du nom (héritage du sniffing pandas → `column_profile`), pour que l'utilisateur sache s'il s'apprête à filtrer un entier, une date ou un texte.
3. **Aperçu live** — bouton « œil » sur le nœud et onglet aperçu dans `BlockEditor` qui affiche les premières lignes du parquet matérialisé.
4. **Profilage de colonne** — cliquer une colonne dans l'aperçu donne ses statistiques (`column_profile` dans `backend/engine.py`).
5. **Validation pré-run en ligne** — un message d'erreur/avertissement visible **sur le nœud** *avant* l'exécution (config incomplète, entrée non branchée…). Critère P7.
6. **État d'erreur sur le nœud** — badge `erreur` après run raté, avec le détail en *tooltip* sur le nœud (P8).
7. **Aide/info-bulles par option** — bulle pédagogique (`InfoBubble`) ou `qb-hint` qui *explique le concept du bloc* dans l'Inspector. (Une bulle décorative `?` du nœud ne compte pas — elle n'expose pas les options.)
8. **Exemples / modèles** — gestes d'amorçage : préréglages cliquables, exemples insérables, testeur peuplé d'exemples.
9. **Retour sur gros volumes** — affordance pour la grande taille : cache, dry-run sans matérialisation, badge de progrès, drapeau « tronqué », warning sur cellules excessives.
10. **Chaînes i18n centralisées** — un *catalogue* (clé → libellé) plutôt que des libellés FR codés en dur.

Légende : ✅ couvert ; ⚠️ partiel / inégal ; ❌ absent ; n/a sans objet pour ce bloc.

## Lecture verticale (par capacité)

- **Sélecteur de colonnes typé (col. 2).** Le typage est **inégal** : il s'affiche dans `ColChecklist` (`Inspector.jsx:253-266`) → utilisé en Pivot, Dedup, Cols, Clean/Groupe ; il s'affiche dans `QueryBuilder.jsx:42` pour SQL ; il s'affiche en `title=` sur les *checklists* « keys » de Validation/Analyse (`Inspector.jsx:1150`). Il **ne s'affiche pas** dans `ColSelect` (`Inspector.jsx:242-251`) — utilisé pour Pivot.`pivot_column/value_column`, Validation.`target_column`, Analyse.`column`, Filtre (deux selects parallèles), Calc *coltag*. Conclusion : la version *typée* existe et est partiellement réutilisée, mais le sélecteur *mono-colonne* reste non typé par défaut.
- **Validation pré-run en ligne (col. 5).** Aucun nœud n'affiche d'état d'erreur **avant** l'exécution. Tout ce qui ressemble à de la validation in-canvas est en réalité (a) un état post-run (badge `erreur`/`warn`), ou (b) le dry-run *de Validation* qui vit dans l'Inspector, pas sur le nœud (`routing.jsx:20-37`). Aucun autre bloc n'a d'équivalent.
- **État d'erreur sur le nœud (col. 6).** **Universellement couvert** par `StatusBadge` (`SourceNode.jsx:36-41`) pour les nœuds qui utilisent `NodeFooter` (Source, SQL, Union, Pivot, Calc, Filter, Cols, Report), et par des `node-foot` ré-écrits avec le même motif pour Validate (`ValidateNode.jsx:51-55`), Dedup (`DedupNode.jsx:44`), Clean (`CleanNode.jsx:27-29`), Export (`ExportNode.jsx:24-30`). Export va plus loin : badges `désactivé` et `vide — rien écrit` métier-spécifiques. P8 honoré post-run.
- **Aide/info-bulles par option (col. 7).** Le motif `InfoBubble` (`Inspector.jsx:1351`) est repris dans Validate, Split, Filter (×2), Cols, Report, Clean/Groupe, Calc, Union, SQL, Dedup — **mais pas dans Pivot, ni dans Source, ni dans Export** (qui passent par `qb-hint` ou `title=`). Pivot est l'écart le plus marqué : c'est un concept Excel non trivial sans bulle.
- **Exemples / modèles (col. 8).** **Un seul bloc** dispose d'exemples cliquables : Calc (`CALC_EXAMPLES`, `Inspector.jsx:888-895`). Validation a un *testeur* avec `test_samples` persistés (`Inspector.jsx:96`, `engine.py`) — affordance différente mais voisine. Tous les autres : zéro.
- **Retour sur gros volumes (col. 9).** Trois affordances ciblées : (a) cache Source + bouton ⟳, (b) `route_preview` / `split_scan` côté Validation, (c) cap à 200 valeurs distinctes + `truncated:true` pour Analyse. **Aucune** affordance générique : pas de barre « ce bloc est lourd », pas d'avertissement avant un Pivot sur 200 k lignes, pas de mémoire d'export. La barre de progrès SSE (P8 — voir guide) supplée *partiellement* au runtime.
- **Chaînes i18n centralisées (col. 10).** **Absentes partout** — la recherche `i18n|t(|translate|locale` dans le front ne renvoie aucune infra. Les libellés FR sont codés en dur dans chaque composant. C'est la colonne « ❌ uniforme ».

## Dissymétries les plus criantes

1. **Validation vs Union.** Validation embarque **~1 297 LOC** dédiées (Inspector, helpers, routing) avec dry-run, testeur, sorties dynamiques, conditions DNF, masques, contrôles de groupe, `split_scan`. Union se contente de **2 cases à cocher** dans **22 LOC d'Inspector** (`Inspector.jsx:902-924`) et **26 LOC de nœud** (`nodes/UnionNode.jsx`), alors que l'union est aussi piégeuse (divergence de schémas par nom vs par position) : pas d'aperçu des schémas qui *vont* s'aligner, pas de mise en garde sur les colonnes absentes d'un côté. C'est l'écart de profondeur le plus visible du produit — le contre-exemple direct de P10.
2. **Pivot orphelin.** Pivot est le seul bloc « concept-lourd » sans **aucune `InfoBubble`** dans son configurateur (`Inspector.jsx:268-323`) — un `qb-hint` d'une ligne en guise d'explication. À la fois aucun exemple, aucun typage du sélecteur de pivot, aucune aide sur le mode « unpivot ». Dissymétrie avec Calc et Validation qui *sur-expliquent*.
3. **Calc seul à offrir des exemples.** `CALC_EXAMPLES` (`Inspector.jsx:888-895`) est un geste pédagogique fort, isolé. SQL, Clean, Filter, Pivot — tous des blocs où un préréglage rendrait service — n'en ont aucun. La promesse implicite du bouton « Inspector » varie donc d'un bloc à l'autre : sur Calc on apprend, ailleurs on est seul.
4. **Filtre non typé alors qu'il compare.** `FilterConfig` (`Inspector.jsx:991-1004`) propose deux *selects* parallèles mainCols/refCols **sans afficher le type**, alors que la sémantique « comparer une clé composite » repose entièrement sur la concordance de type. C'est la seule fonctionnalité où l'absence de typage trahit directement le contrat du bloc.
5. **Pré-run universellement absent.** L'écart structurel : *aucun* nœud n'affiche d'état problématique sur le canevas tant qu'on n'a pas cliqué « Exécuter ». La seule exception (le dry-run de Validation) reste cachée dans l'éditeur. C'est une dissymétrie *entre le réel et le contrat de P7* plus qu'entre les blocs, mais elle pèse autant sur les 12 blocs exécutables : `engine.py` lève des erreurs claires (l. 323, 364, 1147, 1404, 1449, 1675) qui pourraient toutes être détectées statiquement et remontées sur le nœud avant le run.
